# Feedify Firestore Schema Documentation

## Overview
This document describes the complete Firestore database schema for the Feedify application, including all collections, documents, and fields.

## Collections

### 1. `users` Collection
Stores user profile information and preferences.

**Document ID**: `userId` (Firebase Auth UID)

**Fields**:
```javascript
{
  userId: string,                    // Firebase Auth UID (document ID)
  email: string,                     // User's email address
  displayName: string,               // User's display name
  photoURL: string,                  // Profile picture URL (optional)
  phone: string,                     // Contact phone number (optional)
  location: string,                  // User's location (optional)
  bio: string,                       // Short bio/description (optional)
  bookmarks: array<string>,          // Array of bookmarked listing IDs
  preferences: object,               // User preferences object
  createdAt: timestamp,              // Account creation timestamp
  updatedAt: timestamp,              // Last profile update timestamp
}
```

**Indexes**: 
- Single field: `email` (Ascending)
- Single field: `createdAt` (Descending)

**Use Cases**:
- User profile management
- Bookmarking listings
- User preferences storage
- User rating and review aggregation

---

### 2. `listings` Collection
Stores all food listings posted by users.

**Document ID**: Auto-generated by Firestore (addDoc)

**Fields**:
```javascript
{
  // Core Information
  title: string,                     // Food item name/title (required)
  description: string,               // Detailed description (required)
  quantity: string,                  // Amount available (e.g., "5 portions")
  foodType: string,                  // Category (cooked, fresh, bakery, packaged)
  
  // Location Data
  location: string,                  // Text address/location name
  coordinates: {
    latitude: number,                // GPS latitude
    longitude: number,               // GPS longitude
  },
  
  // Time Information
  pickupTime: string,                // Preferred pickup time window
  expiryTime: timestamp,             // When food expires/is no longer available (optional)
  
  // User Information
  userId: string,                    // Lister's Firebase UID (required)
  hostName: string,                  // Lister's display name
  contactInfo: string,               // Phone number or contact method
  
  // Status & Claim Information
  status: enum,                      // "available" or "claimed"
  claimedBy: string | null,          // Claimer's userId (null if available)
  claimerName: string | null,        // Claimer's display name
  claimerEmail: string | null,       // Claimer's email
  claimerPhone: string | null,       // Claimer's phone
  claimedAt: timestamp | null,       // When the food was claimed
  
  // Images
  images: array<string>,             // Array of image URLs (optional)
  
  // Metadata
  createdAt: timestamp,              // When listing was created
  updatedAt: timestamp,              // Last update timestamp
}
```

**Indexes**:
- Composite: `status` (Ascending) + `createdAt` (Descending)
- Composite: `userId` (Ascending) + `createdAt` (Descending)
- Composite: `claimedBy` (Ascending) + `claimedAt` (Descending)
- Single field: `createdAt` (Descending)
- Single field: `status` (Ascending)

**Use Cases**:
- Browse all available listings
- Get user's own listings
- Get user's claimed listings
- Filter by status
- Real-time updates with onSnapshot

### 3. `notifications` Collection
Stores notifications for users about food listings.

**Document ID**: Auto-generated by Firestore

**Fields**:
```javascript
{
  userId: string,                    // Recipient user ID
  senderId: string,                  // Sender user ID
  type: enum,                        // "claim", "listing_near", "message"
  title: string,                     // Notification title
  message: string,                   // Notification message
  listingId: string,                 // Related listing ID (optional)
  
  read: boolean,                     // Has user read this notification
  readAt: timestamp,                 // When notification was read (optional)
  
  data: object,                      // Additional context data
  createdAt: timestamp,              // When notification was created
}
```

**Indexes**:
- Composite: `userId` (Ascending) + `createdAt` (Descending)
- Composite: `userId` (Ascending) + `read` (Ascending)

**Use Cases**:
- Send notifications to users
- Fetch user's notifications
- Mark notifications as read

---

### 4. `messages` Collection
Stores direct messages between users.

**Document ID**: Auto-generated by Firestore

**Fields**:
```javascript
{
  senderId: string,                  // Sender's user ID
  receiverId: string,                // Receiver's user ID
  senderName: string,                // Sender's display name
  
  message: string,                   // Message content
  listingId: string,                 // Related listing ID (optional)
  
  read: boolean,                     // Has receiver read this message
  readAt: timestamp,                 // When message was read (optional)
  
  createdAt: timestamp,              // When message was sent
}
```

**Indexes**:
- Composite: `senderId` (Ascending) + `receiverId` (Ascending) + `createdAt` (Descending)
- Composite: `receiverId` (Ascending) + `read` (Ascending)

**Use Cases**:
- Direct messaging between users
- Chat history
- Mark messages as read

---

### 5. `userReviews` Collection
Stores ratings and reviews for users.

**Document ID**: Auto-generated by Firestore

**Fields**:
```javascript
{
  raterId: string,                   // User giving the rating
  ratedUserId: string,               // User being rated
  
  rating: number,                    // Rating score (1-5)
  review: string,                    // Review text
  
  listingId: string,                 // Related listing ID (optional)
  
  createdAt: timestamp,              // When review was created
}
```

**Indexes**:
- Single field: `ratedUserId` (Ascending)
- Composite: `ratedUserId` (Ascending) + `createdAt` (Descending)

**Use Cases**:
- Rate users after transactions
- Display user reputation
- Calculate average ratings

## Data Types Reference

| Type | Description | Firestore Type |
|------|-------------|-----------------|
| `string` | Text data | String |
| `number` | Numeric value | Number |
| `boolean` | True/false | Boolean |
| `timestamp` | Date and time | Timestamp |
| `array<T>` | List of items | Array |
| `object` | Key-value pairs | Map |
| `enum` | One of predefined values | String |
| `null` | Empty/no value | Null |

## Security Rules

See `FIRESTORE_SECURITY_RULES.md` for complete security rules implementation.

### Key Principles:
- **Authenticated Users Only**: Most write operations require authentication
- **User Own Data**: Users can only edit their own profiles and listings
- **Public Read**: Listings can be read by anyone
- **Owner Deletion**: Only listing creators can delete listings

---

## Best Practices for Development

1. **Always use timestamps** for `createdAt` and `updatedAt` using `Timestamp.now()`
2. **Index composite queries** before using them in firestore
3. **Use batch operations** for multiple updates
4. **Validate data** on client before sending to Firestore
5. **Use TypeScript types** for data objects in your React components
6. **Clean up listeners** by unsubscribing from onSnapshot()
7. **Handle errors** gracefully with try-catch blocks

## Migration & Maintenance

### Adding New Fields
1. Add field with default value to document create functions
2. Update TypeScript interfaces
3. Handle null/undefined in read operations
4. Test with existing documents

### Deleting Collections
1. Delete in reverse dependency order
2. Backup data first to Cloud Storage
3. Update security rules first
4. Test in staging environment first
